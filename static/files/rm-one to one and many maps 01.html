CONVERSELY

but…
ExtinguishOldFrom – is optimisable in the backward link table since c – is going to be a necessary backpointer, in other words c – a is guaranteed.
Thus…
Arguably, there is no point attempting to optimise the removal of links, since the backpointers are not optimisable in one case and forward pointers are not
optimisable in the other case. In other words, any optimisation on an operation in one table (be it forward or backward links) will not echo in the mirror table.
However...if we are still doggedly keen…
One could doggedly still attempt optimisation by doing it where possible i.e. not doing the wiring in one table, but doing it in the other. Is this nuance worth it?
The major hit is the lookup so if we could skip the lookup then it might be worth it.
Well, the
FindObjectPointingToMe
FindObjectPointedToByMe
calls still need to be made within ExtinguishOldTo and ExtinguishOldFrom, but the calls to
forwardLinks.GetToObjs(fromObj, relId);
backLinks.GetToObjs(toObj, relId);
within RelationshipOperation(opRemove) could be optimised such that only one or the other is ever needed, but never both (in the case of one to one relId that
is). This would save us one lookup (from links via relids to toobjs).
Ironically we would potentially need to add another lookup into the constraint table, since we need to do the optimising only in the case of one to one relIds.
However we could avoid this lookup by passing a flag all the way from
RemoveExistingRelationships()
where the constraint lookup is already done. The cost of this is extra parameter on the Extinguish…() methods and also on the RemoveRelationship() method.
